A Product is fundamentally a rule set and a contract factory. The rule set defines what constitutes a valid contract instance and how contract obligations are structured. The factory generates binding contracts when instantiated.

A Product has: a name, a globally unique immutable id, a version (major.minor.patch semver), and belongs to a version set (also named and uniquely identified). Each Product belongs to a Domain (with the same identity and versioning semantics).

A Product Definition specifies the Schemas, Views, Actor-Roles, Processes, Interactions, Schedules, and Presentations required to satisfy the Product's Objectives. These objectives are expressed in business outcome terms (contracts produced, obligations satisfied).

Every Product operates within constraining contracts: legal frameworks, regulatory principles, policies, rules, shareholder promises, Board strategies, and internal policies. These constraining contracts are themselves generated by other Products, creating a recursive dependency network.

A Product has lifecycle states (Draft, Active, Deprecated) that affect its contract generation capability and validity scope.

A 'Manifest' is a machine (and human) readable declaration of the content of a Product, comprising of both direct structured content, and URLs to additional resources. Manifests are shared between system components so the Product-neutral components can use than as run-time configuration artefacts

A 'Product Service' is an executable software service components that exposes a Manifest and supplies URL endpoints for Product dynamic configuration and processing features
